Core feeling.

Milagift should feel like a premium personalized gift shop, not a loud product marketplace.

White minimalismUse white as the page base and light grey blocks for structure. Accent color should support emotion, not decorate every surface.
Personal detailCopy should mention names, photos, dates, flowers, messages, memories and gifting moments.
Square product mediaAll product primary images must be square and fill the container. Use block-level media boxes with aspect-ratio: 1 / 1, no inner padding and object-fit: cover. This applies to collection, search, recommendation, related product and article product cards.
Soft but crispRounded cards and calm surfaces paired with sharp typography, strong CTAs and clear hierarchy.
Real ecommerce copyUse long descriptive product titles like the reference site. Avoid short placeholder names such as “Gift Bag”.

Motion and scroll interaction.

Motion is gentle and useful. It should make modules feel discovered while scrolling, then get out of the way.

Scroll reveal structureEvery major page module should use reveal-section. The first hero may also include is-visible so it does not flash on load. Inner cards can use local stagger only when the module has 3+ repeated items.
Reveal effectInitial state: opacity: 0, filter: blur(10px), transform: translateY(54px) scale(0.985). Visible state removes blur and returns to translateY(0) scale(1).
TimingUse 0.85s cubic-bezier(0.3, 1, 0.3, 1) for opacity, filter and transform. Avoid bouncy easing on page sections; it should feel premium and calm.
Observer triggerUse IntersectionObserver with threshold: 0.12 and rootMargin: "0px 0px -12% 0px". Unobserve after reveal so modules do not animate repeatedly while scrolling back.
Card hoverProduct images scale to 1.035. Category images translate up 4px and rotate -2deg for a playful but restrained feel. Hover should never shift layout.
Button hoverTranslate up 1-2px. Hero CTA reverses white to black on hover; product card CTA uses black background, white text and a reversed arrow dot.
Back to topUse one global .back-to-top button. Keep it hidden on initial viewport, show after min(520px, 72vh) scroll, and call window.scrollTo({ top: 0, behavior: "smooth" }) on click.
CarouselUse transform translate with the same cubic-bezier(0.3, 1, 0.3, 1), around 0.5s. Pause autoplay on hover. Progress bars animate width, not layout-heavy properties.
Reduced motionRespect prefers-reduced-motion: remove animation, remove smooth scroll, set revealed content visible immediately and keep carousels usable without autoplay dependency.

Required page structure

Use the same scroll contract across homepage, product, collection, cart, checkout and account pages. This is the baseline markup pattern:

<main class="page">
  <section class="hero-wrap reveal-section is-visible">...</section>
  <section class="category-band reveal-section">...</section>
  <section class="hot-products reveal-section">...</section>
  <section class="reviews reveal-section">...</section>
</main>

Module application rules

  • Hero: visible immediately; internal text and CTA may have a small load delay, but no large scroll jump.
  • Product grids: reveal the section once; cards can stagger 40-70ms only for first viewport entry.
  • Checkout/account forms: reveal whole panels, not individual fields, so functional pages remain stable.
  • Footer/newsletter: reveal as one quiet block; avoid playful rotations near legal or payment content.

Canonical CSS

.reveal-section {
  opacity: 0;
  filter: blur(10px);
  transform: translateY(54px) scale(0.985);
  transition:
    opacity 0.85s cubic-bezier(0.3, 1, 0.3, 1),
    filter 0.85s cubic-bezier(0.3, 1, 0.3, 1),
    transform 0.85s cubic-bezier(0.3, 1, 0.3, 1);
}

.reveal-section.is-visible {
  opacity: 1;
  filter: blur(0);
  transform: translateY(0) scale(1);
}

Canonical JS

const revealSections = document.querySelectorAll(".reveal-section");

if ("IntersectionObserver" in window) {
  const observer = new IntersectionObserver((entries) => {
    entries.forEach((entry) => {
      if (!entry.isIntersecting) return;
      entry.target.classList.add("is-visible");
      observer.unobserve(entry.target);
    });
  }, { threshold: 0.12, rootMargin: "0px 0px -12% 0px" });

  revealSections.forEach((section) => observer.observe(section));
} else {
  revealSections.forEach((section) => section.classList.add("is-visible"));
}

Scroll rhythm preview

First viewportHero is already visible; first supporting module rises softly.
Shopping modulesCategory bands, product rails and editorial cards reveal once.
Utility modulesForms, order cards and footer use panel-level reveal only.

Mobile and accessibility

  • Keep the same observer, but reduce section spacing before adding more animation.
  • Do not animate fixed header, mobile menu drawer content or payment button position while scrolling.
  • For reduced motion, set .reveal-section to visible, remove transform and remove transition.
  • Never rely on animation to communicate state; loading, success, failure and payment states still need text and icon labels.

P1 checkout

Checkout

checkout
Structure

Checkout is an independent split layout, not a normal theme inner page. Hide the marketing header, offer bar, footer, breadcrumbs, cart hero and page title. Use a full-height left white workflow area and a right light-grey order-summary area.

Components

Do not handwrite a new component system for checkout. The split layout is checkout-specific, but internal parts must reuse design-language components: .btn, .card, .coupon-box, .summary-line, .status-chip, global toast and shared product-card media rules.

Forms

Use the shared field/input language for editable areas. Checkout may be denser than marketing pages, but labels, focus states, borders and radii still follow the global form component instead of raw custom controls.

Summary

Order summary sits inside the right grey column in a white card. Totals use shared summary rows, total amount is orange, coupon uses the shared coupon component, product rows are white cards with square thumbnails, long product titles, variant/customization text, quantity and subtotal rows.

State

Errors are inline and specific. Payment loading should keep the summary visible.
